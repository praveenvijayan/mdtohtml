# Uninstall Ratchet

Remove Ratchet from this repo safely. Removal spans three different things — be
explicit about each.

## 1. Preview

Confirm `.ratchet-install.json` exists at the repo root first — the
uninstaller reads only that file to know what to remove; it never guesses
from filenames. If it's missing (a pre-manifest install, or Ratchet copied in
by hand), `scripts/ratchet-uninstall.sh` dies immediately with a clear
message and removes nothing; tell the user to either reinstall with
`scripts/bootstrap.sh` first (to get a manifest) or remove files by hand.

When the manifest is present, run the teardown in dry-run and show the user
exactly what would be removed: `bash scripts/ratchet-uninstall.sh`. Explain
the three categories:
- **Framework files** — the `installed` paths recorded in
  `.ratchet-install.json` (skills, workflows, scripts, `AGENTS.md`, and any
  other paths the installed profile(s) recorded). Removed by default, **except**
  a path whose on-disk content hash no longer matches the one recorded at
  install/update time — that one is locally modified, so it's **kept** and
  reported, not removed.
- **Generated files** — the `generated` paths recorded in
  `.ratchet-install.json`: `GATES.md`, `memory/`, the skill mirrors
  (`.claude/skills`, `plugin/skills` — byte-identical copies of
  `.agents/skills` regenerated by `setup.sh`), `.env.example`, and
  `.ratchet-version`. **Kept by default** — pass `--purge-memory` to remove
  just `memory/`, or `--purge-generated=path,path` to name any others
  explicitly (e.g. `--purge-generated=GATES.md,.claude/skills`).
- **Your plan issues** — `plan/*.md` is **kept by default**. Ask whether to
  also remove it (`--purge-plans`). `.env` is never removed (it isn't in the
  manifest).

## 2. Confirm, then remove files on a branch

`main` is normally protected, so the removal must go through a PR — do not push
to `main`.

1. `git checkout main && git fetch origin && git pull --ff-only`
2. `git checkout -b ratchet-uninstall`
3. Run with the user's choices, e.g. `bash scripts/ratchet-uninstall.sh --yes`
   (add `--purge-memory` / `--purge-plans` / `--purge-generated=path,path`
   only if the user agreed).
4. `git add -A && git commit -m "Remove Ratchet"` and push the branch.
5. Open a PR and tell the user to review and merge it to complete the file
   removal.

## 3. GitHub-side cleanup (ask per item; never assume)

These are not files and don't go through the PR. Offer each, and act only on an
explicit yes:
- **Labels** — delete the nine `state:*` / `priority:*` labels
  (`gh label delete <name> --yes`). Deleting a label just unlabels issues; it
  does not delete them.
- **Secret / variable** — `gh secret delete FACTORY_PAT`,
  `gh variable delete RATCHET_AUTO`.
- **Branches** — delete `ratchet/planning` and any merged `agent/issue-*`
  branches (`git push origin --delete <branch>`).
- **Open planning PR** — offer to close it (`gh pr close`).

## Hard rules

- **Never delete issues.** They are the user's work items, not Ratchet's.
- **Never remove branch protection.** It's the user's safety setting — only
  remind them it exists if they want it gone.
- Preserve `generated` paths (`GATES.md`, `memory/`, skill mirrors,
  `.env.example`, `.ratchet-version`) and `plan/*.md` unless the user
  explicitly opts to purge them.
- Never remove a path that isn't recorded in `.ratchet-install.json` — the
  manifest, not a filename heuristic, decides what's Ratchet's. No manifest,
  no removal.
- A framework path whose content hash has diverged from the recorded one
  (locally modified) is always kept, never force-removed.
- Do file removal on a branch via PR (main is protected); only the GitHub-side
  cleanup happens immediately, and only with per-item confirmation.
